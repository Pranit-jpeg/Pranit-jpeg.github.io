A local machine for filtering noise.

JobPipeline is a small, opinionated piece of research infrastructure that reads the job market for me. Scrapers, a SQLite queue, a Claude scoring prompt, and a threshold. That’s the whole thing.

Pranit Choudhary · Boston, April 2026

The problem I set out to solve was not “find me a job.” The problem was that every tool that claims to find you a job first requires you to wade through several hundred postings that aren’t for you, and ends with you doubting the entire enterprise. I wanted the filter before the firehose, not after.

So I built one. JobPipeline is a Python program that runs once a day on my laptop. It hits the career pages of companies I’m actually willing to work for, writes the postings to a local SQLite database, scores each posting against four tailored résumés using the Claude API, and silently deletes everything that doesn’t clear a 75 % match. What’s left is a small Kanban board I can actually read over coffee.

The architecture, more or less

Four phases, one cron job. The design priority was reproducibility — every run produces a dated log, every job carries its scrape date, its score, its scoring model, and the résumé variant it was matched against.

┌─ phase 1 ────────────────┐ ┌─ phase 2 ──────────────┐ │ scrapers/* │ │ score_jobs.py │ │ 185 company pages │ → │ Claude · 4 résumés │ │ SQLite dedup by URL │ │ returns 0–100 % + DQ │ └──────────────────────────┘ └────────────────────────┘ │ │ ▼ ▼ ┌─ phase 3 ────────────────┐ ┌─ phase 4 ──────────────┐ │ drop if score < 75 % │ │ h1b_sponsors.py │ │ (no prompt, no review) │ → │ tag known sponsors │ └──────────────────────────┘ └────────────────────────┘ │ ▼ ┌── Kanban on :5000 ──┐ │ Flask + vanilla JS │ └─────────────────────┘

What each phase actually does

Phase 1 runs the scrapers — one per company, subclassed from a small base that handles retries, rate-limiting, and the company-agnostic normalisation. Each posting is keyed by URL, so re-runs are idempotent. Phase 2 reads the four résumé text files, fetches each unscored job’s page, and asks Claude to return a structured score, a best-matching résumé variant, a one-sentence justification, and a disqualification flag for anything that fails basic eligibility (seniority, location, visa). Phase 3 deletes below-threshold jobs without asking; this is the most useful part of the whole system. Phase 4 tags the survivors with their H-1B sponsorship status from a static list I curate.

“The most useful feature is the one that deletes. Filters that ask you to confirm each deletion are filters you will stop using by week two.”

What it removed, and why that’s the whole point

On today’s run, JobPipeline considered 184 postings. It kept 68. That ratio matters less than what got dropped: the “Senior Data Scientist with 8+ years” postings I am not yet qualified for, the staff-engineer jobs that slipped through a title filter, the roles at non-sponsoring shops that would evaporate my candidacy on first contact with HR. The pipeline saves me roughly ninety minutes of reading a week, and a much less quantifiable amount of morale.

What I’d still like to fix

  • The scorer occasionally over-weights keyword matching when the job description is short — I want to add a length penalty.
  • H-1B status is currently a static list of known sponsors. It should at least warn when a job posting mentions sponsorship in the body text.
  • The dashboard is a single-page Flask + vanilla JS app. It deserves a rewrite in a real framework, and yet it never gets one.
  • I want per-résumé throughput tracking — which variant is surviving the threshold most often, and what that implies about positioning.

Why this is on a portfolio site

Because it’s the project I’d want someone hiring me to look at. Not for the software — the software is fine — but for the habit it encodes: build the instrument before you do the measurement. Write the pipeline before you trust the number. I do the same thing with a panel regression. I do it with a policy review. It’s the same move.
